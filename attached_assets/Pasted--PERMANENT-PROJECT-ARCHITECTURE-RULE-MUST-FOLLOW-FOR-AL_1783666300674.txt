## PERMANENT PROJECT ARCHITECTURE RULE — MUST FOLLOW FOR ALL FUTURE WORK

I want you to inspect, enforce, and permanently document the architecture of this project.

This is a mandatory project rule and must be followed for every future feature, fix, refactor, integration, deployment, and database change.

### CORE ARCHITECTURE

The required architecture is:

* Replit = development and building environment only
* Supabase = complete production backend and source of truth
* Vercel = complete production frontend hosting and production runtime
* GitHub = source code repository and deployment source where applicable

Replit must NOT become a required production dependency.

The live business must continue operating even if:

* the Replit project is stopped
* the Replit project is deleted
* the Replit account is unavailable
* the project is transferred to a buyer without the original Replit workspace

---

## 1. SUPABASE MUST OWN THE BACKEND

Audit the entire project and ensure all persistent production backend functionality is correctly hosted through Supabase where appropriate.

This includes:

* PostgreSQL database
* database tables
* authentication
* user accounts
* user profiles
* Row Level Security policies
* database functions
* database triggers
* storage buckets
* uploaded files
* uploaded images
* persistent application data
* business data
* admin data
* customer data
* application records
* production state
* server-side database logic where appropriate
* Supabase Edge Functions where backend server-side execution is required and suitable

Supabase must remain the single source of truth for persistent production data.

Do not store production data permanently inside:

* Replit filesystem
* local JSON files
* SQLite inside Replit
* Replit database products
* Replit-specific storage
* temporary local files
* browser localStorage where the data should actually be persistent server-side business data

localStorage may only be used for genuinely non-critical client preferences where appropriate, never as the authoritative source of business or customer data.

---

## 2. VERCEL MUST OWN THE LIVE FRONTEND

The production website/application must run from Vercel.

Ensure:

* the frontend builds correctly on Vercel
* production routes work on Vercel
* refreshes on nested routes work correctly
* required environment variables are configured for Vercel
* no frontend feature depends on Replit being online
* no production URL points to a Replit development server
* no production API request accidentally calls localhost or a Replit URL
* no assets required in production are served only from Replit
* no hidden Replit runtime dependency exists

Where serverless production runtime logic is required, use the correct production architecture deliberately. Do not silently create a dependency on Replit.

---

## 3. REPLIT IS THE BUILDER ONLY

Treat Replit as:

* coding environment
* AI development assistant
* testing environment
* debugging environment
* temporary development preview

Do NOT treat Replit as:

* production database
* production storage
* required live backend
* required production API host
* permanent production server
* source of persistent customer data
* dependency required for the live business to function

Any Replit-specific development configuration must remain development-only.

* localhost URLs

## 5. ENVIRONMENT VARIABLE RULES

Audit all environment variables and classify them correctly.

### Frontend-safe variables

Only public/client-safe variables may be exposed to browser code.

Examples:

* Supabase project URL
* Supabase anon/publishable key
* explicitly public configuration values

### Secret server-side variables

Never expose secrets to frontend/browser code.

Examples:

* Supabase service_role key
* private API keys
* webhook secrets
* admin secrets
* signing secrets
* private third-party credentials

Ensure secret keys are only used in secure server-side environments.

Never place the Supabase service_role key in:

* VITE_ variables
* NEXT_PUBLIC_ variables
* frontend source code
* browser bundles
* localStorage
* public GitHub code

If secure server-side logic is needed, implement it using an appropriate production server-side environment such as Supabase Edge Functions or properly designed Vercel server-side functions, depending on the architecture and requirement.

---

## 6. FUTURE DEVELOPMENT RULE

From now on, before implementing any new feature, determine where it belongs:

### Frontend/UI

Place in the frontend codebase and ensure it deploys to Vercel.

### Persistent data

Store in Supabase PostgreSQL.

### Authentication

Use Supabase Auth.

### File/image storage

Use Supabase Storage unless explicitly instructed otherwise.

### Database permissions

Use Supabase RLS.

### Database automation

Use Supabase functions/triggers where appropriate.

### Secure backend execution

Use an appropriate production server-side environment, preferably Supabase Edge Functions when closely tied to Supabase/backend operations, or Vercel server-side functions where architecturally appropriate.

### Development/testing

Replit may be used temporarily, but must not become a production dependency.
